# 技术决策记录

## 决策 1：先做本地相册，再做 AI 修图

原因：

- 商业级修图难度高。
- 本地媒体库是所有功能的地基。
- 导入、扫描、缩略图、搜索跑通后，AI 和修图才有稳定依托。

## 决策 2：本地优先

原因：

- 用户照片、家庭影像、人脸数据和企业素材高度敏感。
- 本地优先是本产品与云相册的重要差异。
- 企业用户更容易接受私有化和本地处理。

## 决策 3：阶段 2 不接 SQLite 和 AI

原因：

- 当前目标是先让桌面应用能启动。
- 过早接入复杂模块会增加调试难度。
- 小白验收需要每一步清晰可见。

## 决策 4：使用 Tauri + React + TypeScript

原因：

- Tauri 适合本地桌面应用。
- React + TypeScript 适合快速构建复杂 UI。
- Rust 后续适合处理本地文件、性能任务和安全边界。

## 决策 5：不做复杂云同步作为 MVP

原因：

- 云同步涉及账号、权限、加密、成本和安全合规。
- MVP 的核心闭环是本地导入、整理、搜索和基础修图。
- 云能力后续作为可选增强。

## 决策 6：阶段 4 使用纯本地 EXIF 读取

原因：

- EXIF 属于照片隐私数据，必须坚持本地读取和本地入库。
- `kamadak-exif` 是纯 Rust 轻量库，不需要外部程序和云服务。
- 阶段 4 只做基础元数据，不做地图、不做人脸识别，降低隐私和合规风险。

## 决策 7：P0 Windows 试用版先用本地资源预览

原因：

- Windows 测试版必须先跑通“导入文件夹 → 看到数量 → 看到真实照片”的最小闭环。
- jpg/jpeg/png/webp/gif/bmp 由本机 WebView 直接预览，不上传照片，不复制、不删除、不移动、不重命名原图。
- HEIC、RAW 和视频暂不做解码，界面明确提示“已导入，暂不支持预览”，避免用户误以为导入失败。
- 后续阶段 5 再做正式缩略图生成和缓存，降低本次 P0 热修的风险。
